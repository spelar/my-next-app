# AI 에이전트 기반 개발 도입기 (회고)

> 13년차 프론트엔드 팀 리드로서, AI 코딩 에이전트(Claude Code)를 **개인 도구가 아닌 팀 표준**으로 도입하며 정리한 회고입니다.
> 회사 내부 지표·시스템은 밝히지 않고, **방법론과 판단**에 집중합니다. 각 원칙이 실제로 동작하는 모습은 이 저장소(**spelar**) 자체가 근거입니다.

---

## 1. 배경 — 왜 도입했나

여러 세대의 스택이 공존하는 팀(최신 Next.js부터 수년째 운영 중인 레거시 SPA까지)에서 반복적으로 병목이 된 지점은 코드를 "짜는" 속도 자체가 아니었다.

- **보일러플레이트·반복 작업**이 실질 개발 시간을 잠식
- **테스트가 없는 코드베이스**가 많아, 변경 시 회귀 위험이 개발 속도를 억눌렀다
- 신규 합류자의 **온보딩·컨텍스트 파악**에 드는 비용이 컸다

AI 에이전트는 이 세 가지를 동시에 완화할 수 있는 지렛대였다. 다만 "각자 알아서 쓰기"로는 품질 편차와 컨텍스트 단절이 생긴다고 판단했고, **팀 차원의 표준·가드레일과 함께** 도입하기로 했다.

---

## 2. 접근 — 무엇을, 어떻게

### 원칙
1. **컨텍스트가 프롬프트보다 중요하다** — 프로젝트 규칙·구조·관례를 문서(마크다운)로 명문화해 에이전트가 매번 같은 맥락에서 작업하게 했다.
2. **AI 산출물은 반드시 검증을 통과한다** — 생성된 코드는 테스트와 CI 게이트를 통과해야만 병합된다.
3. **도구 도입 ≠ 문화 정착** — 라이선스만 주는 게 아니라, 코드리뷰 기준과 온보딩 절차에 녹였다.

### 핵심 결정 (ADR 요약)

#### ADR-1. AI 에이전트를 "조직 표준"으로 도입
- **맥락**: 개인이 각자 도구를 쓰면 산출물 품질이 들쭉날쭉하고, 프로젝트 컨텍스트가 공유되지 않는다.
- **결정**: 조직 차원 라이선스를 승인받아 팀 전체가 동일한 도구·규칙을 사용. 공통 컨텍스트 문서를 저장소에 두고 함께 관리.
- **결과**: 확산 속도와 산출물 일관성이 올라갔다. 비용이 발생하지만, 절감된 개발 시간이 이를 상회한다고 판단.

#### ADR-2. AI 산출물에 "테스트 게이트" 의무화
- **맥락**: AI가 만든 코드는 그럴듯해 보이지만, 검증 없이는 신뢰할 수 없다. 특히 테스트가 없던 코드베이스에서는 회귀를 잡을 안전망이 없다.
- **결정**: 생성 코드는 **테스트를 동반**하고, **CI에서 타입체크·테스트를 통과해야만 배포**되도록 파이프라인을 구성. QA에게 받은 테스트케이스(TC)를 통합 테스트로 자동화.
- **결과**: 테스트가 전무하던 영역에 안전망이 생겼고, "AI가 빠르게 짠 코드"가 "믿고 배포할 수 있는 코드"가 됐다.

---

## 3. 결과

**정량**
- 기능 개발 **리드타임 대폭 단축**
- 테스트가 없던 코드베이스에 **유의미한 커버리지 확보**
- QA 테스트케이스 기반 **통합 테스트 자동화**

**정성**
- 온보딩·컨텍스트 파악 속도 향상
- 문서화가 개발 흐름의 일부로 자리 잡음
- 운영 이슈 대응 속도 개선

---

## 4. 안티패턴 / 배운 점 (리드 관점)

> 이 절이 이 문서의 핵심이다. 도구를 쓰는 것보다, **어떻게 팀에 안착시키는가**가 리드의 일이었다.

- **검증 책임은 사람에게 남는다.** 에이전트는 초안을 빠르게 만들지만, "맞는지"를 판단하는 건 사람이다. 테스트 게이트가 없으면 속도는 부채로 돌아온다.
- **프롬프트보다 컨텍스트 관리가 품질을 좌우한다.** 좋은 결과는 좋은 프롬프트가 아니라, 잘 정리된 프로젝트 컨텍스트에서 나온다.
- **도구를 주는 것과 문화를 바꾸는 것은 다르다.** 리뷰 기준·온보딩·공통 규칙이 없으면 확산되지 않는다.
- **AI는 시니어의 판단을 대체하지 않고 증폭한다.** 아키텍처 결정, 트레이드오프, 우선순위는 여전히 사람의 몫이며, 에이전트는 그 판단을 더 빠르게 실행하게 해줄 뿐이다.

## 5. 실증 — 이 저장소(spelar)에 같은 방법론을 적용했다

이 포트폴리오는 **동일한 방식으로** 개발·배포됐고, 아래는 저장소에서 직접 확인할 수 있는 사실이다.

| 원칙 | 이 저장소에서의 실제 적용 |
|---|---|
| 컨텍스트 문서화 | 프로젝트 규칙·구조를 문서로 관리하며 에이전트와 협업 |
| 테스트 동반 | **Vitest 3 + Testing Library + MSW**로 **7개 파일·19개 테스트** (회사 표준과 동일 계열 스택) |
| CI 게이트 | GitHub Actions에서 **타입체크 + 테스트 통과 시에만 빌드·배포** (`test` job → `build-and-deploy`) |
| 안전한 아키텍처 | 외부 API 키를 서버 라우트로 프록시해 클라이언트 비노출, 비밀번호 bcrypt 해싱 |
| 접근성 | 자동완성 검색을 ARIA combobox 패턴(키보드 내비게이션·`aria-activedescendant`)으로 구현 |
| 무중단 배포 | Docker + Nginx **Blue/Green** + Let's Encrypt HTTPS |
